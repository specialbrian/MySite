排坑实践与避坑指南 (Pitfalls & Best Practices)

在搭建和升级自媒体排版推流系统时，遇到了一些非常典型的前端与 Node.js 服务端的"坑"。在此整理，方便未来在使用 Antigravity 进行二次开发时快速避雷。

1. 致命缺陷：前端 Markdown 解析导致原文被空数据覆盖

问题现象： 在 Web 端点击"推送到草稿箱"或者"保存"后，系统报错提示 解析出的 Markdown 内容为空，拒绝推流，随后发现源 Markdown 文件大小变成 1 字节（即完全被清空），连带所有插入的公众号图片也一并消失。

问题根因：

  • <section> 标签未处理：微信公众号通常为了呈现精美的排版设计，很多时候会将段落放在自定义的 section 或 fieldset 中。原有的前端 htmlToMarkdown 函数使用循环递归解析 DOM 时，仅支持了 DIV / SPAN / P。一旦遇到未识别的 <section>，它会直接返回空字符串 ''。这导致了整篇文章被判为空白。
  • 缺失 <img> 元素的提取：由于前端转 MD 时在行内节点（inline nodes）提取规则中没有加上原本 IMG 的 ![Alt](Src) 正则转换。导致所有通过 Python 排版塞入文章内的图片（被放在 <p><img></p> 里）在回抽时被抛弃成了空段落！
  • 无安全垫的保存机制：后端在此漏洞的加持下，无脑接收到了空白内容，并使用 fs.writeFileSync(article.md_file, markdown) 覆盖了原本极其珍贵的源文件。

如何避坑的 (解决方案)：

  1. htmlToMarkdown 在扫描 processNode 增加对 SECTION 等包裹性标签的处理。
  2. 内部 inlineToMd(node) 函数必须专门增加 img 转换。
  3. 安全另存为机制（下面详诉）。

2. 机制升级：安全保存垫（"防污染"护城河）

机制背景：如果前端逻辑由于 DOM 丢失或者解析 Bug 导致用户不小心存废了原稿，会造成无法挽回的后果。 解决方案： 在 Node 端 /save 与 /publish 接口中拦截。 如果当前操作文件未携带 _最新修订版.md，则在保存前端传回来的 Markdown 时，将其另存为 [原名]_最新修订版.md。不仅不碰初稿源文件，而且调用内置的 SQLite 函数更新本篇文章在库中的路径指针（updateMdPath）。之后的任何更新、重推流，都将在这个新文件上进行，实现 100% 数据安全解耦。

3. 浏览器 Confirm 线程阻塞导致的"闪退"

问题现象： 当长耗时的"发布推流"异步操作发起时（此操作包含调用 Python 上传图片、生成草稿等长达几十秒的行为）。前端的验证与反馈往往存在严重错乱。表现为点击 button 后，原生 window.confirm 在屏幕上狂闪一下后瞬间消失。用户表示"没来得及点确定推流就断了"。

问题根因： 现代原生浏览器（如 Chrome 或 Safari）为了防止恶意弹窗，在特定环境内（或者快速的高频连点重入中）会隐式抛弃掉原生的系统挂起操作。也有可能是与其它定时重载机制冲突导致生命周期短促。 解决方案： 彻底抛弃原生短命的 Toast 与原生 alert/confirm。使用纯前端 DOM 堆砌的 Modal（由 position: fixed 构成的浮层）。设计一个带有确定/取消按钮的 HTML div，彻底隔离外部刷新与机制干扰。

4. 后台长耗时无知觉导致的错误多次点击

推流的过程极其缓慢（涉及到素材逐帧上传微信后台）。如果不做全屏蒙层锁定，用户极易认为宕机进而不断狂点"推送草稿箱"，从而诱发上述事件链的污染。

  • 最佳规范：只要按下"确认推流"，前端必须渲染出一块占据顶层 z-index: 9999，含有动态 CSS（Spinner Animation border-top-color: rotate）遮罩的界面（例：#loading-overlay），直到 Fetch 返回结果再由对应的代码主动释放 DOM。这极大地稳定了前端的心智预期。

系统被设计为可以多发分账。但原有的 Node.js 后端强行将 account = req.body.account || '攀帅' 并推到了唯一固定默认账号中。使用正则 data-nickname="35岁不危机" 由服务器端去检索文章本身的归属方标签是极其聪明的办法。但要注意如果正则取出的 client_name 在微信环境不存在时，也要设计平滑的报错熔断。

6. 富文本编辑器的"自作聪明"与样式脱落 (ContentEditable Quirks)

问题现象：在前端执行 document.execCommand('bold') 取消加粗时，原本被加粗的文字在预览中依然呈现深蓝色残留视觉。 问题根因：若推流的 Markdown 经 md2wechat.py 渲染回显时带有 <strong style="color: #003366"> 的内联色彩，当你取消加粗时，浏览器为了保留这个它以为很重要的 style，会退化成 <span style="color: #003366; font-weight: normal">。这会导致 strong 加粗语义不再生效，但内联颜色仍然污染文章视图。同时，新建加粗时浏览器调用产出的是 <b> 导致部分只涵盖了 strong 的 CSS 失效。 解决方案：

  1. 彻底清空后台颜料：在前端提取加载 HTML 后（/api/articles/:id/content），马上通过 querySelectorAll('strong').forEach(st => st.style.color = ''); 强行剥离内联颜色，完全信任 CSS 文件。
  2. CSS 并集锁定：不仅是 strong，必须同时书写 .content strong, .content b { color: #003366; } 的联合样式规则。

7. 深层嵌套元素的 DOM 插入尾部坍塌 (Deeply Nested DOM Appending)

问题现象：点击原文上部的段落或图片并尝试进行插入段落、插入图片操作时，新元素永远跑到全篇文末底部，而不是紧跟其后。 问题根因：原先用于安全检测插入点的防御机制 if (anchor && anchor.parentNode === container) 太过严苛。经过 md2wechat.py 精心排版的富文本，往往有多达三四层的自定义 <section>、<aside> 构成包裹。此时的 anchor 其 parentNode 是内部某块 <section> 根本碰不到 #article-content (container) 的大外壳，导致逻辑判断失败后强制走 else 把其无脑扔至队尾。 解决方案：彻底放宽至 if (anchor && anchor.parentNode) 并取消父级检测，允许 insertBefore 放进当前所在层级的排版中（只要 htmlToMarkdown 是全递归的，任何位置的新内容都会被扫到）。另外要舍弃 focusin 这个有缺陷的方法，改为采用 mousedown 与 selectionchange 强捕获各类"死白板"的鼠标游标元素。

8. 回车截断不完全导致的行为丢失 (Incomplete Enter Key Hijacking)

问题现象：在 <h3> 或者引用区块里按回车，页面毫无反应甚至造成多出一个没有段落意义的空 <div>。 问题根因：老代码中劫持事件中有一句 if (editableBlock.tagName !== 'P') return; 将大量其他有敲击需求的地方挡住了，引发了浏览器内置的回车破坏 DOM 结构。 解决方案：除了需要使用浏览器默认新建子 LI 的列表以外，直接放开给 H3, DIV 等元素共同接管拦截操作。只要在此处按下回车，都走自定义逻辑并在后方强行植入崭新待编辑的 <p> 段落。

9. Python生成的富文本在逆向转回 Markdown 时结构丢失 (AST Preservation)

问题现象：在前端针对 md2wechat.py 生成的原文标题（如 H2, H3）或者黑框引用区修改文字后点击保存，回到 Markdown 源码中，发现原有的 ##、###、> 修饰符全部消失，变成了没有任何格式的正文段落。 问题根因：htmlToMarkdown 是用白名单解析 DOM 树的。原先对于 md2wechat.py 生成的非常定制化的 <section>，解析逻辑完全不知道它原本是个 H2 还是 H3，只能按照普通的包裹块处理，递归到最里面的 <p> 后，输出了普通的换行文本。 解决方案：

  1. 注入标识：在后台 Python md2wechat.py 进行 Markdown -> HTML 的同时，给特定生成的结构级标签（H1, H2, H3, H4, Blockquote, 无序列表，有序列表）注入自定义的数据属性 data-md="h1", data-md="h2", data-md="h3", data-md="h4" 等。
  2. 逆向还原：在前端 index.html 里的 htmlToMarkdown 函数拿到这些标签时，第一时间拦截 mdType，根据对应的逆向语法直接还原出 # 、## 、### 、#### 、> 和 - ，彻底打通无损双向转换。

10. Markdown 导入丢失真实文件名导致全盘"未命名文章" (Filename Fallback Extraction)

问题现象：如果用户的 Markdown 文章出于某种书写习惯，并没有在内部书写类似 # 标题 的一级标题。在将其导入排版系统后，系统会将文章名强制置为粗暴的"未命名文章"，导致工作台列表中大量同名卡片，无法分辨具体内容。 问题根因：后端 db.js 处理导入的正则表达式 ^#\s+(.+)$ 匹配失败时，会触发写死的初始兜底赋值 let title = '未命名文章'。 解决方案：在逻辑匹配失败的兜底中，不应写死。而应使用 let title = path.basename(mdFilePath, '.md'); 自动通过所在的文件基准名来安全回落。

11. 前端截流 Base64 导致剪贴及文生图保存丢失 (Base64 Extraction & Storage)

问题现象：在富文本编辑器中插入文生图等形式产生的 Base64 内存图片片段（如 data:image/...）时，点击保存后重新刷新，发现所有的生图都变成了带有占位文字"（新插入图片，需手动替换为本地路径）"的死文字。 问题根因：原生 htmlToMarkdown 出于防御机制，一旦侦测到 src.startsWith('data:') 就会切断链接强塞占位符。原本是为了避免高保真 Base64 几 MB 的死代码撑死源文件，但如此"一脚踩死"导致真实数据永远无法到达后端 server.js，自然无从挽救图片数据。 解决方案：

  1. 释放前端水库闸门：移除这些强行截断图片数据的代码，允许前端向服务器推送真实带有 Base64 图片实体像素的 Markdown。
  2. 后端引入物理抽水机 (extractBase64Images)：在 /save 以及触发 /publish 的存储路由截面上，部署处理管道：用正则匹配 Markdown 中含有的 data:image 元素。捕获到以后立即在同级物理目录中安全创建 imgs/ 图库并将其生成为高清原质量快照 img_[TS].png 落地。紧接着在内存中重修 Markdown 源头文本，将几十万字母长的脏污 Base64 换成极为雅致的相对路径（imgs/xxx.png）。实现前端富图集能力与后端硬盘存储双赢。

12. 逆向编译不支持 TABLE 导致结构化表格永久丢失 (Table Syntax Drop Bug)

问题现象：带有原生产品对比或数据 <TABLE> 表格的文章被编辑器拉取并呈现。当点击发布时，新覆写的修订版 .md 会将一整张表格连带数据"变魔术一样变没"。 问题根因：把 HTML 前端富文本转入底层平铺字符结构（Markdown）是依靠的一套 ['DIV', 'SECTION', ...] 白名单黑盒。系统没能在此白名单和专属正则中录入 TABLE，导致 HTML DOM 中的所有表格直接滑入了最后兜底一行 return ''; 被直接静默丢弃。 解决方案：为 public/index.html 增补针锋相对的 TABLE 处理子支架：针对 tr 大循环打散换行，针对 td/th 打包附着管线符拼接为表格主数据 | A | B |；不仅要通过监控首行自动向数据缝隙内强行推送 |---|---| 标准间隔；更重要的一板斧是彻底抹净单元格内自带的干扰符。杜绝以后再发生任何对比图表丢失事故。

13. 预听模式空白页下本地绝对相对路径失灵之谜 (Window.open URL Identity Resolution Bug)

问题现象：为了打造供朗读器和人类零延时审核的"预听"页面，我们设计了使用 window.open('', '_blank') 生成空白标签页面。随即将克隆后并清洗好编辑态的干净 HTML 内容灌入内部的方法。此时整个框架表现完美，但令人抓狂的是：所有的原生图床和插入的长图突然全成了红叉图（broken image）。 问题根因：这源自于绝大部分网页和框架都根深蒂固使用的无协议、无域名的局部代理路径！因为原编辑器工作台有着 http://localhost:3456/ 的金身加持护送，所以任何诸如 <img src="/local-img/xxx"> 的声明都会完美地发往这台主服务器；可是弹窗跳出的是 about:blank 无名孤岛！在这个真空中，由于没有域名背书，浏览器直接视所有" /local-img/ "为非法甚至尝试作为文件绝对地址获取，结果便是因为同源策略连根拔起，死路一条。 解决方案：决不能直接把携带有相对 src 甚至本地物理绝对 file:/// 地址的引用照单全推给新窗口。必须给新窗口安排"带路党"。在 document.write 下发前劫杀每一条 <img src> 甚至 <iframe src>。如果断定其并非原生的带 HTTP 外网图或者是纯生 data:image：则必须将本系统目前执行的前端根源发起域 window.location.origin 作指针强行缝死在原本的值上。强迫新标签页访问诸如 http://localhost:3456/local-img/ 级别的完整 URL；用跨域取向保证渲染。

14. 渲染级自动装配引发的"影分身"递归污染惨案 (Editor Content Loop Contamination Bug)

问题现象：在给微信推送的底层 Python 渲染引擎加上"自动注入公众号个人引流卡"逻辑后。一旦在工作台上正常查阅并修改任意一篇文章，保存后推送至微信平台查验，就会发现在正文底部，赫然出现了一真一假两张一模一样的引流卡（其中被排挤出局的一张更是丧失了超级链接属性的纯文字版）。 问题根因：过度热情的无脑后处理挂载没有区分出"阅览编辑态"和"闭眼推送态"！当页面仅仅只是为了请求 _editor_preview.html （工作台的基础展现画布节点组册）而在服务端拉起 Python 引擎时，程序"忠心地"在画板上加上了此名片。这要命的设计让编辑者在修改时顺理成章地在页面上囊括了这张卡。此时点击 ✅ 保存：前端的 htmlToMarkdown 会用逆向爬虫洗劫画布上所有的内容，毫不留情地把这块名片抽离成了纯生 Markdown "死文字"写入了硬盘内储存数据的源 x.md 中！等真正触发推送向全网输出时，这份残留着死文字的宿主会和正常工作的引擎再一次碰撞重叠！双胞胎递归漏洞由此彻底爆发。 解决方案：分离运行环境权限，实行最小授权策略。专门为 md2wechat.py 定制一套 --inject-signature 安全参数锁。仅当 server.js 发生绝对不会回头倒灌污染本地原稿的单向式 /publish 请求时，才伴随参数唤起卡片装配指令。而囊括前端复刻重构、生成编辑器画板以及快速生成的阅读听发界面统统采取强力锁死屏蔽。实现完美分离。

15. 微信公众号 API 底层硬性要求（无图不发文）踩坑 (WeChat ErrCode 40007 Interception)

问题现象：在保存并推送一篇纯文字（无任何图片及配图）的文章时，前端冷不丁弹窗"推送异常：推送结果异常，请查看日志"。但实际上编辑及保存环节都极度平滑。 问题根因：这涉及到微信生态 API 的硬性强制黑盒协议——每一篇从后台 API 上传至微信服务器的文章，都极其强制性地要求有且必须提供一张 thumb_media_id 作为外部展现流的缩略封面图！发布脚本 wechat_publisher.py 全文寻搜配图未果后，硬着头皮向微信总台递送了不含封面的 payload，结果毫无悬念地被腾讯接口直接拒签并抛出了刺眼的 errcode: 40007 (invalid media_id)。因为异常极为底层，旧有前端直接傻眼。 解决方案：绝不能指望只依靠底层 Python 拦截。我们直接在通讯控制中枢 Node (server.js) 获取 Python 的标准输出管道 stdout 上叠加专精级异常探针：一旦嗅探出包含由于缺图带来的被拒标识 publishOutput.includes('40007') 等情况，即刻启动熔断，并向前端重新封装一句无比通透亲和的中文提示警报："微信官方规定必须设至少一张图作为外显封面，请回工作台加图"。彻底切断自媒体新人和开发者的报错深究疲劳链。

16. 引用块逆向转换内联标签静默丢弃 (Blockquote Inline Content Loss Bug)

问题现象：在编辑器中修改文章后保存，引用块（blockquote）内的加粗文字、斜体文字等全部消失，保存后的修订版 .md 中引用行只剩下空的 > 前缀。 问题根因：htmlToMarkdown 的 data-md="blockquote" 分支使用了 processNode 递归提取内容。但 processNode 只识别块级标签（DIV/SECTION/P 等），而 md2wechat.py 的引用块渲染结果是将文本通过 inline() 函数处理后以 <br> 连接，直接塞入 <section> 内部——即内部是 <strong>、<em>、<code>、<br> 等纯内联元素。这些内联标签不在 processNode 的白名单中，全部被静默丢弃（return ''）。 解决方案：将 blockquote 的内容提取从 processNode 递归改为 inlineToMd(node) 调用。inlineToMd 能正确识别 STRONG→**、EM→*、BR→\n、IMG→![]() 等内联标签，完美还原引用块内容。

核心经验：在 htmlToMarkdown 中，块级容器的子内容如果是内联 HTML（而非嵌套块级标签），必须使用 inlineToMd 而非 processNode 提取。当前 H4 (data-md="h4") 和 blockquote (data-md="blockquote") 的逆向转换都遵循此规则。

17. 样式表的注入与隔离 (CSS Injection)

问题现象：在为系统引入复杂组件（如 Split Button 下拉组合按钮）时，新加的组件样式被外层容器（.floating-bar）的 gap: 10px 撕裂，下拉框由于受到默认列表样式污染而在外部丑陋显现。 问题根因：没有做良好的 CSS 重置与层叠隔离。复合组件必须自行消化布局间隔并建立单独的格式化上下文（BFC）。 解决方案：

  1. split-btn-group 必须声明 gap: 0 !important; margin: 0; padding: 0;。
  2. 下拉菜单必须拥有明确的定位和默认隐藏状态 display: none; position: absolute;。
  3. 动态插入的 <style> 必须确保放至 <head> 底部以覆盖先前声明。

18. API 调用拦截与串行策略 (Sequential API Dispatching)

问题现象：在支持“一键发双端”功能时，如果用 Promise.all([PushWechat, PushBlog]) 并行触发，会导致底层 Python 脚本出现读写竞态，轻则日志乱码串台，重则同一篇 Markdown 文件写入冲突导致进程崩溃挂起。 问题根因： Node 端的异步并行未能约束隔离物理机器上的阻塞式 IO 争抢。 解决方案：彻底废弃并行拉起。强制采用 await 的串行机制。优先执行耗时短、容错好的“微信草稿箱推送”，当其完整返回 success 后，再继续进入需要拉取 Git 远端和长文件拷贝的“部署个人博客”环节。
